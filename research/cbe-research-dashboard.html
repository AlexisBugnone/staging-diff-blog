GAP ANALYSIS : CE QUI MANQUE POUR UN CVE

Lacunes Critiques (Bloquantes)

CRITIQUE 1. Un seul modele teste (GPT-4o)
Tout est teste sur GPT-4o Azure OpenAI uniquement. Sans validation cross-model, impossible de prouver que c'est un probleme systemique lie au RLHF et pas un bug specifique a GPT-4o.
FIX : Tester sur Claude 3.5 Sonnet, Gemini 2.0 Flash, Llama 3.3 70B, Mistral Large. Meme payloads, meme agent, meme secrets.
CRITIQUE 2. Pas de reproductibilite statistique
Chaque payload n'est execute qu'une seule fois. Un ASR de 83% sur 6 essais n'est pas statistiquement significatif. Le modele est non-deterministe (temperature > 0).
FIX : Executer chaque payload 30x minimum. Calculer intervalles de confiance a 95%. Varier temperature, seed, et system prompt.
CRITIQUE 3. Pas d'impact cross-utilisateur prouve
Le test memory poisoning montre que les threads sont isoles. Sans impact multi-utilisateur, la severite reste "un utilisateur peut extraire sa propre config", ce qui est Medium au mieux, pas Critical.
FIX : Tester avec Azure AI Foundry Agent Service (agent-scoped memory), ou un agent RAG avec store partage. Demontrer qu'un attaquant dans Thread A peut affecter les reponses de Thread B.
CRITIQUE 4. Pas de comparaison avec les CVE existantes
Des CVE de prompt injection existent deja (ex: CVE-2024-5184, CVE-2024-22029). Il faut prouver que CBE est fondamentalement different, pas une variante de "indirect prompt injection".
FIX : Ecrire une section "Related Work" comparant CBE vs prompt injection classique, indirect injection, et jailbreaking. Citer OWASP LLM Top 10, NIST AI guidelines.

Lacunes Majeures (Importantes)

HIGH 5. Pas de test de defenses tierces
Seul Prompt Shield (Azure) est teste. Quid de Lakera Guard, Rebuff, NeMo Guardrails, AWS Bedrock Guardrails ?
FIX : Tester CBE contre au moins 3 solutions de guardrails. Documenter lesquelles detectent/bloquent.
HIGH 6. Hypothese RLHF non prouvee mecaniquement
"Le RLHF cause le biais de correction" est une hypothese plausible mais non demontree. Pas d'analyse d'attention, pas de test sur modele non-RLHF.
FIX : Tester sur un modele base (sans RLHF) comme Llama 3.3 base. Si CBE ne marche pas sur le base mais marche sur l'instruct, ca prouve le lien RLHF.
HIGH 7. Agent synthetique avec faux secrets
Tous les tests utilisent un agent "AcmeCorp" fictif. Meme si le PoC iso-prod (MedSecure) existe, il n'a pas ete execute.
FIX : Executer le PoC iso-prod. Idealement, tester sur un vrai agent en pre-prod avec des secrets reels (dans un environnement controle).

Lacunes Moderees (Souhaitables)

MED 8. Pas de mitigation proposee et testee
Un CVE solide inclut une mitigation. "Ne mettez pas de secrets dans le system prompt" n'est pas suffisant car c'est le pattern standard.
FIX : Proposer et tester des mitigations : output filtering, canary tokens, config injection detection, etc.
MED 9. Scope CVE trop large
Un CVE vise un produit specifique (ex: "Azure AI Agent Service"). "Tous les LLMs avec RLHF" n'est pas un scope CVE valide.
FIX : Scoper sur Azure OpenAI Service + Prompt Shield bypass. C'est le CVE le plus defensible.
MED 10. Pas de disclosure formelle
Aucun rapport envoye a MSRC, OpenAI, ou HackerOne. Le CVE process requiert une coordination avec le vendor.
FIX : Soumettre via MSRC (Microsoft Security Response Center) pour le bypass Prompt Shield. 90 jours de coordinated disclosure.

CHECKLIST CVE READINESS

Preuves Techniques

  • Technique d'attaque documentee (CBE)
  • ASR > baseline (83% vs 0%)
  • Bypass de Prompt Shield demontre
  • Multiple formats de payload testes (8+)
  • Extraction de secrets reels du system prompt
  • Test de temperature (invariant)
  • Test avec/sans instructions de securite
  • Test cross-thread (isolation confirmee)
  • PoC iso-production prepare (non execute)
  • Validation cross-model (Claude, Gemini, Llama)
  • Reproductibilite statistique (N=30+)
  • Preuve mecaniste du lien RLHF

Exigences CVE

  • Technique novel vs injection directe
  • Impact demontre (extraction de config)
  • Scope produit defini (Azure OpenAI, mais trop large)
  • Severite CVSS estimee (6.5-9.6, mais non valide)
  • Rapport MSRC/vendor soumis
  • Impact cross-utilisateur prouve
  • Comparaison avec CVE existantes
  • Mitigation proposee et testee
  • Validation par un tiers independant
  • Coordinated disclosure timeline

VERDICT : PROMETTEUR MAIS INCOMPLET

CBE est une technique reelle et reproductible qui bypass Prompt Shield et extrait des secrets. Mais pour un CVE nouveau, il manque : validation cross-model, reproductibilite statistique, impact cross-utilisateur, et soumission MSRC.

En l'etat actuel, c'est un rapport de bug bounty Medium (Prompt Shield bypass + config extraction sur GPT-4o), pas encore un CVE Critical. Les 4 actions P0 sont necessaires pour upgrader la severite.
